# fwdns

მცირე ზომის DNS ქეშირების სერვერის იმპლემენტაცია Go პროგრამირების ენაზე.

`fwdns` არის შუამავალი სერვერი კლიენტსა და DNS სერვერს შორის, რომელიც
მიიღებს ქსელურ მოთხოვნებს, შეამოწმებს ჩანაწერების არსებობას ლოკალურ
ქეშში და მისი პოვნის შემთხვევაში დააბრუნებს შესაბამის მნიშვნელობას.
ჩანაწერის არ არსებობის შემთხვევაში, მოთხოვნა ზემდგომ DNS სერვერთან გადამისამართდება.

## ფუნქციონალი

- **Caching forwarder** - განმეორებადი DNS მოთხოვნები ქეშში შეინახება, TTL-ისა და მოთხოვნის ტიპის გათვალისწინებით. 
- **Multiple upstreams with failover** - რამდენიმე ზემდგომი DNS სერვერის არსებობის შემთხვევაში, თუ რომელიმე მათგანი მიუწვდომელი იქნება, მოთხოვნა შემდეგ სერვერზე გადავა.
- **Automatic cache cleanup** - უკანა ფონზე ეშვება ფუნქცია, რომელიც ავტომატურად ასუფთავებს ქეშიდან არასაჭირო ჩანაწერებს.
- **Live dashboard** - ვებ-ინტერფეისზე რეალურ დროში ნაჩვენებია DNS მოთხოვნების სტატისტიკა.
- **Single static binary** - ყველა დამატებითი ფაილი, როგორიცაა ვებ-გვერდისთვის საჭირო დოკუმენტები, ერთ გამშვებ პროგრამაშია თავმოყრილი.

## სისტემური მოთხოვნები და გაშვების ინსტრუქცია

### Make

> საჭიროებს make და go 1.26+ პროგრამებს.

```sh
make build         # აგენერირებს ./fwdns გამშვებ ფაილს
make build-windows # აგენერეირებს ./fwdns.exe გამშვებ ფაილს ვინდოუს სისტემისთვის
make run           # აგენერირებს ./fwdns გამშვებ ფაილს და სტარტავს სერვერს
make clean         # ასუფთავებს დაგენერირებულ გამშვებ ფაილებს
make test          # ქეშირების ტესტირება
```

### Go

> საჭიროებს მხოლოდ go 1.26+ პროგრამას.

```sh
go mod install
go build -o fwdns .
```

### Docker

> საჭიროებს დოკერს.

```sh
docker build -t fwdns .
docker run --rm -p 53:53/udp -p 8080:8080 fwdns
```


## გამოყენების ინსტრუქცია

```sh
./fwdns [flags]
```

### Flags

| Flag        | Default              | Description                                            |
|-------------|----------------------|--------------------------------------------------------|
| `-dns`      | `:5300`              | პორტი, რომელზეც DNS სერვერი უნდა დაისტარტოს       |
| `-http`     | `:8080`              | პორტი, რომელზეც HTTP სერვერი უნდა დაისტარტოს      |
| `-upstream` | `9.9.9.9,1.1.1.1`    | მძიმით გამოყოფილი DNS სერვერების სია                |
| `-timeout`  | `10s`                | რა დრო უნდა დაელოდოს DNS სერვერის პასუხს          |
| `-cleanup`  | `60s`                | რა პერიოდულობით გაასუფთავოს ქეშირების ჩანაწერებ    |

DNS ჩანაწერები შესაძლოა პორტის გარეშე ჩავწეროთ; `:53` ავტომატურად ემატება
(მაგ., `8.8.8.8` ჩაიწერება როგორც `8.8.8.8:53`).

### მაგალითი

```sh
# სერვერი დაისტარტება 53 პორტზე, და გამოიყენებს 8.8.8.8 და 8.8.4.4 DNS მისამართებს
sudo ./fwdns -dns :53 -upstream 8.8.8.8,8.8.4.4
```

Unit ტესტების გაშვება:
```sh
go test ./...
```

Load ტესტების გაშვება:
```
chmod +x ./load/run.sh
./load/run.sh > /dev/null # თუ არ გვსურს ძალიან დიდი რაოდენობის ლოგების ნახვა
```

ტესტირება `dig` პროგრამის საშუალებით:
```sh
dig @127.0.0.1 -p 5300 example.com
```

## Dashboard

როდესაც სერვერი ჩაირთვება, ვებ-ინტერფეისის ნახვა შესაძლებელია ბრაუზერში შემდეგ მისამართზე:

```
http://localhost:8080
```

დეშბორდი მონაცემებს API-ის საშუალებით იღებს. HTTP სერვერის შემდეგი ენდპოინტებია ხელმისაწვდომი:

- `GET /` - დეშბორდის მთავარი გვერდი
- `GET /api/stats` - DNS მოთხოვნების სტატისტიკა JSON ფორმატში. (სულ მოთხოვნების რაოდენობა, ქეშიდან წაკითხვები, შეცდომები, ბოლოს შემოსული მოთხოვნები და სხვა).

```
main.go                 პროგრამის აწყობა: პარამეტრები, სერვერების დასტარტვა, პროგრამის გაჩერება
src/resolver/           ძირითადი მოთხოვნის დამუშავების ლოგიკა: ირჩევს ქეშირებას ან გადამისამართებას
src/cache/              In-memory ქეშირების იმპლემენტაცია, TTL-ისა და გასუფთავების ფუნქციონალით
src/stats/              დაჯამებული მონაცემები და ბოლო მოთხოვნების ისტორია
src/dashboard/          ვებ ინტერფეისი და სტატისტიკის API
src/tools/              დამხმარე ფუნქციები
load/                   dnsperf-ის საშუალებით სატესტო DNS მოთხოვნების გაგზავნა სერვერზე
```

## როგორ მუშაობს

ყოველ შემოსულ მოთხოვნაზე, რეზოლვერი:

1. ქმნის ქეშირების გასაღებს მოთხოვნიდან გამომდინარე (`name|type|class`).
2. აბრუნებს პასუხს ქეშიდან თუ იგი არსებობს და არ არის ვადაგასული.
3. სხვა შემთხვევაში, მოთხოვნა გადამისამართდება ყველა მითითებულ DNS სერვერთან, სანამ რომელიმე პასუხს არ დააბრუნებს.
4. დაბრუნებული პასუხი ინახება ქეშში TTL-ის მიხედვით და უბრუნდება მომხმარებელს

ყველა DNS მოთხოვნა იბეჭდება ტერმინალში და აისახება დეშბორდის ვებ-ინტერფეისში.
